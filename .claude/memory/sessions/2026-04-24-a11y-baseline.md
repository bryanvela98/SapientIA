## Static keyboard / a11y audit (what code guarantees)
Grepped for the common offenders ‚Äî all clean:

- ‚úÖ **No positive `tabindex`** anywhere in `src/`. `grep -rn tabindex
  src/` returns nothing.
- ‚úÖ **Every `<Button>` has text or an `asChild` Link with text.** No
  icon-only buttons without labels. (DebugPanel's Copy button uses
  text, not an icon.)
- ‚úÖ **All inputs have `<Label htmlFor>` pairing:** `topic-input`,
  `message`, `pacing-select`, every `RadioGroupItem` in Onboarding,
  the DebugPanel Switch (via `useId()`).
- ‚úÖ **`aria-describedby` wired** on the composer (help text) and
  on each radio option (visible description beneath).
- ‚úÖ **`aria-busy` toggles** on the Send button during streaming,
  on the Save button during onboarding submit, and on the Start
  button during session creation.
- ‚úÖ **`role="log"` + `aria-label`** on the transcript `<section>`;
  screen-reader users can jump there via the rotor.
- ‚úÖ **`role="status" aria-live="polite" aria-atomic="false"`** on
  the live assistant bubble so appending text is announced
  incrementally (aligns with ADR-018 ‚Äî when we upgrade to token
  streaming, this attribute remains correct).
- ‚úÖ **`role="alert"`** on inline error text (onboarding + chat).
- ‚úÖ **`<fieldset>` + `<legend>`** per profile dimension in
  onboarding; semantic grouping rather than divs.

## First VoiceOver sanity pass ‚Äî what to verify manually
I can't run VoiceOver from here. The four targets from the plan,
with what SHOULD announce based on the ARIA in place:

1. **Landing on `/onboarding`**, VO should announce the page title
   "SapientIA ‚Äî inclusive Socratic tutor" (window title) and reach
   the first `<legend>` ("Visual") when Tab enters the fieldset.
2. **Tabbing through the form**, each control should announce
   "Visual, radio group, none, 3 of 3" or similar ‚Äî shadcn's
   RadioGroup renders as `role="radiogroup"` with proper item
   state. Descriptions under each option are tied via
   `aria-describedby`, so VO reads label + state + description.
3. **On `/chat`**, after sending a message, VO should announce the
   live assistant bubble text because it's inside
   `role="status" aria-live="polite"`. With our current granularity
   (one `tool_decision` per swap), VO will announce the whole
   question in a single burst ‚Äî fine as a baseline; token-level
   announce arrives with Day 4.
4. **Submit buttons should announce "busy"** via `aria-busy={true}`
   while streaming. VO tends to phrase this as "Send, busy,
   button" (varies by version).

10-minute VO budget means: exercise the four targets above, fix
any `aria-*` or semantic-HTML bug that shows up, don't sink hours
tuning. Heavy screen-reader polish is explicitly Day 4.
