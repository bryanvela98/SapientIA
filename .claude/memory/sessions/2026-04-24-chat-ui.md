## Streaming flow (happy path)
```
submit → setMessage('') + setPendingUser(outgoing) + setStreaming(true)
       → startLiveTurn(nextTurnNumber)
for await ev of streamTurn(sessionId, outgoing):
  tool_decision → applyDecision(ev) → live.text updates, decisions pushed
  concept_earned → addEarned(ev) → EarnedFlash badge appears
  concept_told   → addTold(ev) → EarnedFlash badge (outline variant)
  error          → setError(ev.message)
  turn_start/turn_end → no UI effect directly
finally:
  endLiveTurn(); setStreaming(false); setPendingUser(null);
  refresh via getSessionTurns + getSessionState;
  composerRef.current?.focus();
```
Refresh-after-turn is the safety net: if the optimistic user bubble or
live bubble ever get out of sync with the server, the next `getSessionTurns`
overwrites everything from the DB and UI converges.

## Accessibility wiring (baseline)
- Transcript: `role="log"` + `aria-label` so a screen-reader user can jump
  to it via the rotor.
- Live bubble: `aria-live="polite"` + `aria-atomic="false"` — browsers
  append-announce only new content. Our current granularity is one
  tool_decision per update, so users hear the whole tutor question in one
  burst (ADR-018 — token streaming upgrade is a Day 4 task).
- Composer: `<label for="message">` paired to the textarea; `<Input>`/
  `<Textarea>` from shadcn preserves focus-visible rings.
- Focus management: after each turn, composer regains focus in the
  `finally` block, so keyboard users can immediately reply without
  hunting.
- Optimistic bubble is not announced separately (it inherits the log
  context) — avoids double-announcing the same message.

## Broke / gotchas
- **StrictMode double-mount** fires the hydration effect twice in dev.
  Guarded by comparing `sessionId` against the store's current
  `storeSessionId` — the second run short-circuits because the first
  already set it.
- **Scroll-into-view races StrictMode**. The `useEffect` on `turns.length`
  + `pendingUser` handles it with `block: 'end'`, which is idempotent.
- **`ul > li > li`** — when mapping turns, I wrap each bubble component
  in an `<li>`, but `UserBubble`/`AssistantBubble` also return `<li>`
  internally. Cleaned up: they return `<li>` and the outer `<ul>` maps
  directly without an extra wrapper. `LiveAssistantBubble` and
  `pendingUser`'s `UserBubble` also sit inside the same `<ul>`.
- **`topic` empty at first paint**: during hydration we temporarily set
  the store's topic to `''` and backfill from `/state`. The header shows
  "Loading…" for that window instead of an empty string.
