## Broke / gotchas
- **Bug 1 (fixed)**: the plan's `loop.py` appended plain-text learner replies after assistant turns that contained `tool_use` blocks. The API rejects this — each `tool_use` needs a matching `tool_result` in the next user message. Added `_build_user_turn` to wrap the learner reply in tool_result blocks (learner text rides on the primary, non-bookkeeping tool; auxiliary tools get `"ok"`).
- **Bug 2 (fixed)**: the model sometimes emits *only* `mark_concept_earned` with no teaching move, leaving the primary tool with no `question`/`hint`/`answer` and crashing the learner simulator with empty content. Added a chain-through: if only bookkeeping tools fired, immediately re-call the model for a teaching move before returning from `run_turn`.
- **Model ID**: plan said `claude-opus-4-6`; I used `claude-opus-4-7` (latest; matches the "Build With Opus 4.7" hackathon). Logged as ADR-006.

## Open questions surfaced
- The simulated learner (Haiku) is *too* cooperative — it always reasons through hints and almost never stays stuck. Real users will get stuck. Day 2+ should add a harder "stubborn learner" persona to stress-test the hint escalation ladder (levels 1→2→3).
- Ratio 1.00 on the first try is suspicious. Earned-vs-told needs a harder test: (a) learner who refuses to reason, (b) topic the learner actively gets wrong and must be corrected. Without these we don't know whether `deliver_answer` ever triggers correctly.
- `mark_concept_earned` fired *a lot* (often paired with every question). Is the threshold too loose? Should the tutor wait for a check_understanding pass before marking?
